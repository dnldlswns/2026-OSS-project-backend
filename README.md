## 🔀 PR (Pull Request) Convention

### 1. PR 제목 규칙
- feat: 기능 추가
- fix: 버그 수정
- docs: 문서 수정
- style: 코드 스타일 변경 (공백, 세미콜론 등)
- refactor: 코드 리팩토링 (기능 변화 없음)
- chore: 설정, 빌드 관련 변경

예시:
feat: 로그인 API 구현
fix: 회원가입 오류 수정

---

### 2. PR 내용 작성

- 변경 사항:
  - 무엇을 수정/추가했는지 작성

- 작업 이유:
  - 왜 이 작업을 했는지 설명

- 테스트 방법:
  - 어떻게 테스트했는지 작성

---

### 3. PR 규칙

- main 브랜치 직접 push 금지
- feature 브랜치에서 작업 후 PR 생성
- 최소 1명 이상 리뷰 후 merge

---

### 4. 브랜치 규칙

- main: 배포용
- feature/기능명: 개발용

예:
feature/login
feature/signup
